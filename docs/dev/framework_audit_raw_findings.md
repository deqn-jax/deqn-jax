## evaluate.py + irf.py + benchmark.py

**Summary:** evaluate.py is a 1022-line monolith whose four simulation primitives (euler_equation_errors, simulated_moments, stability_check, market_clearing) each hand-roll their own JIT'd simulate-step closures instead of reusing the single, purpose-built training/shocks.py:simulation_step helper that episode.py already uses. This duplication has already produced a real correctness bug: only euler_equation_errors visits disaster states, while simulated_moments and stability_check silently run disaster-calibrated models on a normal-shock-only path. The discrete-chain and disaster special-casing is model-agnostic in mechanism but leaks disaster-specific knowledge (p_disaster constant name, "eq11 in the disaster model" comments) into the generic framework. irf.py's NotImplementedError for discrete chains is a legitimate, well-documented gap, but it makes compare_to_dynare_irfs crash hard rather than skip for discrete models. benchmark.py is small and healthy except for a hardcoded model choices list that blocks new models from the CLI.


### [eval-01] simulated_moments and stability_check silently skip the disaster branch — inconsistent with euler path and trainer
- **severity:** high → **high**  |  **verdict:** confirmed  |  **category:** correctness  |  **effort:** medium  🟠
- **where:** `src/deqn_jax/evaluate.py:399-424, 525-562`
- **description:** euler_equation_errors was carefully patched (lines 112-120, 217-221) to draw Bernoulli(p_disaster) each period and call step_fn with d_disaster=, so the ergodic accuracy report visits disaster states. But simulated_moments (399-424) and stability_check (525-562) only call _draw_eval_shock (Gaussian/discrete) and a plain _sim_step that calls model.step_fn WITHOUT d_disaster. For a disaster-calibrated model these two primitives therefore simulate a normal-shock-only path: reported ergodic means/stds and the stability verdict exclude the disaster branch entirely. This is exactly the bug the trainer-side shocks.py module docstring (lines 11-17) says it was created to fix, reintroduced in two of the four eval primitives. The 'For the disaster model... eq11' comments and p_disaster handling confirm the intent was full-support evaluation.
- **evidence:** euler path: `d_val = (jax.random.uniform(d_key, (1, 1)) < p_disaster)...; next_state, row, st = _sim_step_with_d(state, shock, d_val)` (220-221). simulated_moments path: `shock = _draw_eval_shock(model, shock_key, state)` then `_sim_step` whose body is `next_state = model.step_fn(state, policy, shock, constants)` (412-414, 404) — no d_disaster, no p_disaster lookup anywhere in the function.
- **suggested fix:** Route all four primitives through training/shocks.py:simulation_step (or a thin single-batch eval wrapper around it), which already composes Gaussian/discrete/disaster dispatch correctly. At minimum, replicate the supports_disaster/p_disaster Bernoulli draw + d_disaster threading from euler_equation_errors into simulated_moments and stability_check.
- **verifier reasoning:** Verified at the cited lines. euler_equation_errors detects supports_disaster (line 119) and p_disaster (line 120) and, when p_disaster>0, draws a per-period Bernoulli (line 220 `d_val = (jax.random.uniform(d_key,(1,1)) < p_disaster)`) and calls `_sim_step_with_d(state, shock, d_val)` (line 221), which threads d_disaster into step_fn (line 159). simulated_moments (368-424) and stability_check (493-562) do NOT: both call only `_draw_eval_shock` (lines 412, 539) and a `_sim_step` whose body is `next_state = model.step_fn(state, policy, shock, constants)` (lines 404, 530) with no d_disaster kwarg and no p_disaster lookup anywhere in either function. I confirmed `_draw_eval_shock` (45-62) handles only the discrete-chain and Gaussian cases and never draws a disaster indicator. The disaster step_fn defaults `d_disaster=0.0` (dynamics.py:14), so omitting the kwarg silently simulates a no-disaster path. The disaster model ships with sweep configs setting p_disaster=0.005/0.05/0.1 (e.g. configs/disaster_p005.yaml:18), so this fires in real research runs. The trainer-side rollout does draw the Bernoulli (shocks.py maybe_draw_disaster, 107-134), so the ergodic state distribution the network was trained against includes disaster states; moments/stability reporting under-samples exactly those states, and the stability verdict (stable iff max_ss_deviation<5 etc., line 578) never sees a disaster jump. This is a genuine cross-primitive inconsistency. Note one nuance the finding overstates: the trainer LOSS combines branches analytically as a mixture (loss.py:223-227), not via Bernoulli — so the precise statement 'euler matches the trainer' is about the rollout/state-visitation, not the loss expectation. Severity high is justified for the audit goal (correctness of the headline accuracy/stability report on the flagship model).
- **fix assessment:** Sound in direction. The minimal fix (replicate the supports_disaster/p_disaster Bernoulli + d_disaster threading from euler into the moments and stability _sim_step closures) is correct and low-risk. The 'route all four through shocks.py:simulation_step' option is cleaner but note simulation_step returns (next_state, shock) and does its own shock draw + policy eval — moments needs the policy out and stability needs the policy too, so a thin eval wrapper returning (next_state, policy, shock) would be required rather than calling simulation_step verbatim. Either path works; prefer the wrapper to also fix eval-02.

### [eval-02] Eval primitives re-implement training/shocks.py:simulation_step instead of reusing it
- **severity:** high → **medium**  |  **verdict:** partially-confirmed  |  **category:** duplication  |  **effort:** medium  🟡
- **where:** `src/deqn_jax/evaluate.py:32-62, 133-207, 399-405, 525-531`
- **description:** training/shocks.py exists specifically as the single source of truth for shock dispatch across rollout, eval, and IRF (see its module docstring lines 1-29). episode.py:simulate_step delegates to it. But evaluate.py instead defines its own _model_uses_discrete_chain (32-42, duplicating shocks.py logic and loss.py:299-301), its own _draw_eval_shock (45-62), and five separate JIT step closures (_sim_step_no_d, _sim_step_with_d, _sim_step_discrete, plus two more in simulated_moments/stability_check) that each re-derive the Gaussian/discrete/disaster branching. This is the proximate cause of finding eval-01: the logic drifted because it was copied rather than shared. Adding a new model with any non-Gaussian shock structure now requires editing this branching in N places.
- **evidence:** shocks.py docstring: 'This module provides one place where all three concerns are handled' (lines 19-28). evaluate.py:54 `if _model_uses_discrete_chain(model):` and 60 `return draw_discrete_shocks(...)` re-derive shocks.py:simulation_step lines 164-169. The discrete residual-expectation closure (192-207) duplicates the enumeration logic in loss.py compute_loss discrete branch (loss.py:299-314).
- **suggested fix:** Extract a single eval_simulation_step(model, policy_fn, state, key) into training/shocks.py (or have euler/moments/stability call simulation_step with batch=1). Keep the residual-expectation enumeration (the one genuinely eval-specific piece) as a separate composable function. Delete _model_uses_discrete_chain and reuse shocks.py's check.
- **verifier reasoning:** The factual core is correct: evaluate.py defines its own `_model_uses_discrete_chain` (32-42), its own `_draw_eval_shock` (45-62), and three step closures inside euler plus two more in simulated_moments/stability_check, none of which call shocks.py:simulation_step. episode.py DOES delegate to shocks.py:simulation_step (episode.py:48, 94, 210), so shocks.py is the single source of truth for the rollout but not for eval. The duplication is real and is the proximate cause of eval-01's drift. HOWEVER the 'high' severity is inflated. (a) `_model_uses_discrete_chain` is a 2-line getattr check, not meaningful logic duplication, and is documented as deliberately mirroring the trainer check. (b) simulation_step cannot be reused as-is for eval: it returns (next_state, shock) and draws its own shock internally, but the eval primitives need the policy and (for euler) the next_policy + residuals, and euler's discrete path needs the exact-enumeration residual expectation (192-207) which is genuinely eval-specific and has no analogue in simulation_step. So this is not a drop-in dedup; it is a moderate refactor. The coupling/duplication is real but its impact on the three audit goals is medium (it makes adding a new shock structure require N edits and it caused eval-01), not high.
- **fix assessment:** Direction sound but the suggested fix understates the work. 'Have euler/moments/stability call simulation_step with batch=1' will not work directly because simulation_step returns only (next_state, shock) and re-draws the shock; you cannot recover the policy/residuals from it. The realistic fix is to extract a shared single-batch eval step (returns next_state, policy, optionally residuals) in shocks.py and keep the Π-weighted enumeration separate, exactly as the fix's second sentence allows. Deleting `_model_uses_discrete_chain` in favor of an exported shocks.py predicate is fine and harmless.

### [eval-03] euler_equation_errors is a 175-line monolith with five near-duplicate JIT closures

- **where:** `src/deqn_jax/evaluate.py:70-245`
- **description:** A single function defines three inline @eqx.filter_jit closures (_sim_step_no_d 133-151, _sim_step_with_d 153-173, _sim_step_discrete 175-207) that are 90% identical (policy eval, ndim guards, step, next-policy, residual stack), then a Python for-loop with a three-way if/elif/else dispatch (212-236) selecting among them. The only differences are the step_fn call signature and the residual-vs-expectation computation. - **evidence:** The residual stack `jnp.stack([residuals[name][0] if residuals[name].ndim > 0 else residuals[name] for name in eq_names])` appears verbatim at lines 145-150, 167-172, and 199-201.
- **suggested fix:** Factor the policy->step->residual core into one helper parameterized by the step call and the residual reducer (single-sample vs Π-weighted enumeration). Collapse the three closures into one and pick the residual reducer at construction time, mirroring how trainer.py dispatches train-step variants before JIT.
- **verifier reasoning:** euler_equation_errors does span 70-245 (175 lines) and contains three inline @eqx.filter_jit closures (_sim_step_no_d 133-151, _sim_step_with_d 153-173, _sim_step_discrete 175-207) dispatched by a Python if/elif/else in the t-loop (213-225). The residual-stack comprehension is repeated verbatim three times (145-150, 167-172, 199-201). So the monolith and copy-paste claims are accurate. BUT the title is wrong: it says 'five near-duplicate JIT closures' inside this one function — there are THREE; the other two closures the finding counts live in simulated_moments and stability_check, which are separate functions. The finding's own description correctly says 'three'. Severity should be low, not medium: this is readability/maintainability boilerplate, the closures are genuinely small, and the differences (step signature, residual reducer) are real. A mechanical refactor risks the chaotic-training bifurcation the audit warns about. The 175-line count alone does not make this high-impact for the three audit goals.
- **fix assessment:** Acceptable but not urgent. Parameterizing the core by (step-call, residual-reducer) and selecting at construction time mirrors the existing trainer dispatch pattern and would be clean. Caveat: keep the residual-stack reducer and ndim guards byte-identical to avoid perturbing the numerics; the value is purely maintainability, so weigh against regression risk. This should be done together with eval-02 (shared eval step) or not at all — doing it in isolation duplicates effort.

### [irf-01] compare_to_dynare_irfs crashes (not skips) on discrete-chain models because run_irf raises NotImplementedError

- **where:** `src/deqn_jax/irf.py:58-69`
- **description:** run_irf now raises NotImplementedError for any model with transition_matrix set (lines 58-69) — a legitimate, well-documented gap (continuous 1-sigma IRF is undefined on a categorical shock). However evaluate.py:compare_to_dynare_irfs (752-830) calls run_irf/run_girf inside its per-shock loop with only a `try/except FileNotFoundError` guard (786-789). For a discrete model with a Dynare CSV present, the NotImplementedError propagates uncaught and aborts the whole Dynare comparison and the evaluate CLI. run_girf is doubly affected: it calls run_irf twice (235-236), so it inherits the raise. New discrete models therefore cannot run the standard evaluate pipeline end-to-end.
- **evidence:** irf.py:62 `raise NotImplementedError(...)`. evaluate.py:786-793 only catches FileNotFoundError: `try: dyn = load_dynare_irf(...) except FileNotFoundError: shocks_skipped.append(shock); continue` then unconditionally `net = run_irf(...)`.
- **suggested fix:** In compare_to_dynare_irfs, detect discrete chains up front (reuse _model_uses_discrete_chain) and return {'per_shock': {}, 'shocks_skipped': all_shocks} with a reason, OR catch NotImplementedError in the loop. Better long-term: support a target_z forced-trajectory IRF as the docstring promises so discrete models get a real IRF.
- **verifier reasoning:** Verified. run_irf raises NotImplementedError for any model with transition_matrix and z_state_idx set (irf.py:58-69). compare_to_dynare_irfs loops over model.shock_names and guards only `try: dyn = load_dynare_irf(...) except FileNotFoundError` (786-789), then unconditionally calls run_irf (793) or run_girf (791); run_girf calls run_irf twice (irf.py:235-236) so it inherits the raise. For a discrete model with a Dynare CSV present, NotImplementedError propagates uncaught and aborts the whole comparison and the evaluate CLI. The contract gap is real. HOWEVER I downgrade to low: NO in-tree model currently sets transition_matrix to a non-None value (grep of models/ for `transition_matrix=` non-None returns nothing), so this can only bite a future discrete model, and discrete models cannot have Dynare perturbation CSVs anyway (Dynare doesn't solve categorical-chain models), so the FileNotFoundError guard would in practice skip them first — the crash requires someone to manually drop an irf_e_<shock>.csv into the dynare_dir for a discrete model. That is an edge case, not a blocker for the audit's 'add new models' goal in the near term.
- **fix assessment:** Sound and trivial. Detect discrete chains up front via the same predicate and return early with shocks_skipped=all, OR catch NotImplementedError in the loop. Either is correct and low-risk. The longer-term target_z forced-trajectory IRF is the right design but out of scope for a quick fix. Recommend the early-return guard since it gives a clean reason string.

### [eval-05] market_clearing_errors hardcodes burn_in=500 while euler default auto-clamps — short sims silently return one sample

- **where:** `src/deqn_jax/evaluate.py:321-335`
- **description:** euler_equation_errors and simulated_moments default burn_in=None and auto-clamp to min(500, n_periods//5) so short simulations still produce output (lines 97-102, 385-388). But market_clearing_errors keeps a hardcoded `burn_in: int = 500` (line 326) and forwards it to euler_equation_errors. For a short run (e.g. n_periods=600) the euler guard clamps 500 down to n_periods-1=599, leaving a single residual sample, so mean/max market-clearing stats are computed over one point with no warning. The three primitives have inconsistent burn-in contracts.
- **evidence:** `def market_clearing_errors(... burn_in: int = 500)` (321-327) vs euler `if burn_in is None: burn_in = min(500, max(0, n_periods // 5))` (97-98). The euler over-large guard `burn_in = max(0, n_periods - 1)` (102) silently leaves one sample.
- **suggested fix:** Make market_clearing_errors default burn_in=None and pass through to euler_equation_errors so it shares the auto-clamp, or assert burn_in < n_periods with a clear error in all three.
- **verifier reasoning:** Verified. euler_equation_errors and simulated_moments default burn_in=None and auto-clamp (euler 97-98 `burn_in = min(500, max(0, n_periods//5))`; moments 385-386 identical), plus an over-large guard `burn_in = max(0, n_periods-1)` (euler 102, moments 388) that silently leaves one sample. market_clearing_errors keeps a hardcoded `burn_in: int = 500` (line 326) and forwards it to euler_equation_errors (line 335). So for n_periods=600 the euler over-large guard does NOT fire (500<600), leaving 600-500=100 samples — the finding's specific example math is slightly off (it claims it clamps to 599 leaving one sample, but 500<600 so no clamp; you get 100 samples). The single-sample pathology fires only when 500>=n_periods (n_periods<=500), e.g. a 400-period market-clearing run -> clamped to 399 -> one sample, mean/max over one point with no warning. So the inconsistent-contract claim is true and the one-sample silent-degradation is real for short runs; only the worked example's numbers are imprecise. The CLI always passes args.periods (default large) so this rarely fires in practice -> low severity is correct.
- **fix assessment:** Sound. Make market_clearing_errors default burn_in=None and pass through so it shares the auto-clamp; or assert burn_in < n_periods with a clear error across all three. The assert is preferable to silently leaving one sample. No numerical-stability risk here (pure reporting path).

### [irf-02] load_policy_from_checkpoint duplicates config->NetworkConfig parsing that belongs in config.py

- **where:** `src/deqn_jax/irf.py:257-360`
- **description:** load_policy_from_checkpoint manually pulls a dozen network fields out of the raw YAML dict (lines 289-299) and reconstructs NetworkConfig/OptimizerConfig by hand, including re-implementing the optimizer-switch logic (switch_optimizer/switch_episode/switch_lr, lines 317-329) and parsing the episode number out of the checkpoint filename (321-323). - **evidence:** Manual extraction `hidden_sizes = tuple(net_cfg.get("hidden_sizes", [64, 64])); activation = net_cfg.get("activation", "tanh"); ...` (290-299) then `net_config = NetworkConfig(type=net_type, hidden_sizes=hidden_sizes, ...)` (336-347) — re-listing every field by hand rather than `NetworkConfig(**cfg.get('network', {}))`.
- **suggested fix:** Use the existing config loader (load_config / NetworkConfig(**net_cfg) with Pydantic coercion) to build the configs from the saved YAML, instead of field-by-field dict.get. Centralize the optimizer-switch resolution so checkpoint loading and training share one implementation.
- **verifier reasoning:** Verified and the impact is understated, not overstated. load_policy_from_checkpoint manually extracts a fixed subset of network fields (irf.py:289-299: hidden_sizes, activation, activations, init, multi_head, skip_connections, type, history_len, num_heads, n_layers) and rebuilds NetworkConfig listing only those ten fields (336-347). But NetworkConfig (config.py:613+) has MANY more fields that create_train_state actually consumes for residual/disaster networks: init_scale, use_zlb_feature, zlb_feature_kind, kf_names, reparam_q_as_m, reparam_pi_as_kp_inner, reparam_wtilda_as_kw_inner, and output_links. I confirmed trainer.py:196-251 reads init_scale, output_links, use_zlb_feature, kf_names, reparam_q_as_m, reparam_pi_as_kp_inner, reparam_wtilda_as_kw_inner from network_config when building linear_plus_mlp and disaster_policy_net. This is a load-bearing path used by every evaluate/irf invocation, and it directly undermines evaluation of the flagship disaster model with disaster_policy_net. Severity medium (correctness on the eval path) is appropriate; I considered high but the default_output_links fallback plus the fact that the disaster shape-prior flags are typically also encoded in the model defaults reduce the blast radius.
- **fix assessment:** Sound and important. Replace the field-by-field dict.get with `NetworkConfig(**(cfg.get('network', {})))` and let Pydantic v2 coercion/validation do the work — this auto-picks up every current and future field (init_scale, output_links, reparam_*, etc.) and is exactly how training loads it. One caveat: NetworkConfig uses model_config extra='forbid' (config.py:616), so a saved YAML containing a stale/removed key would raise — acceptable and arguably desirable (fail loud), but worth a try/except or key-filtering if backward-compat with old configs is needed. Also centralize the switch_optimizer/switch_episode/ckpt-episode-from-filename logic (317-329) so checkpoint load and training share one implementation. This is the single highest-value fix in this set for the 'add new models / clean eval' audit goal.


## Networks (mlp, lstm, transformer, linear_p

**Summary:** The networks package has a clean, model-agnostic core (common.py, mlp.py) but suffers from a three-way duplication of the residual "BK-linear + MLP-delta" ansatz: LinearPlusMLP (286 lines, canonical), KfAnchoredMLP (237 lines, the "accidental weaker fork"), and the disaster model's DisasterPolicyNet (923 lines) all reimplement the same forward pass and the byte-identical output_links (linear/log/mixed) branching. KfAnchoredMLP is dead weight in practice — no config uses it, only tests reference it, and its sole feature (K/F gauge anchoring) is a strict subset of DisasterPolicyNet's kf_names mask. The generic viz.py also leaks disaster-specific fields (use_zlb_feature, r_lag_idx, r_lb) and imports DisasterPolicyNet directly, breaking the framework/model boundary. Output bounding/normalization helpers in common.py are well-factored and genuinely model-agnostic.

### [loss-01] Discrete-Markov branch and disaster mixture can both fire, double-counting the expectation
- **severity:** high → **low**  |  **verdict:** partially-confirmed  |  **category:** correctness  |  **effort:** small  ⚪
- **where:** `src/deqn_jax/training/loss.py:299-322, 219-228`
- **description:** compute_loss activates the discrete-enumeration branch purely from model struct fields (transition_matrix/z_state_idx present), with NO gate on config.expectation_type. - **suggested fix:** Gate the discrete branch on the resolved expectation_type (thread an explicit `expectation_type` / `use_discrete` flag from the trainer into compute_loss instead of sniffing model struct fields), and add an assert at trainer setup that `transition_matrix is not None` and `p_disaster>0` are not simultaneously active (or define the intended joint semantics explicitly). - **verifier reasoning:** Two distinct claims, only one survives. ### [steadystate-01] Numerical solve_steady_state silently converges to a wrong, under-determined steady state
