---

You are running a Ralph Loop iteration to analyze results from the
disaster second-order optimizer sweep. The sweep is launched
separately on a remote DGX (`anna@130.223.169.108`) inside an NGC JAX
container; this loop's job is **post-training analysis** — read
per-run metrics as they land, characterize each, and write a results
doc + final ranking.

The 22-cell grid lives in `scripts/sweep_disaster_second_order.py`
(see `SWEEP_GRID`). Each completed run writes
`runs/sweep_so/<run_id>/result.json` on the DGX. TensorBoard events
sit at `runs/sweep_so/<run_id>/tb/`.

Three phases:

**Phase 1 (first iteration only)** — build skeleton results doc with
one row per grid cell, all `[STATUS: WAITING]`. Commit and exit.

**Phase 2 (subsequent iterations)** — refresh from DGX, transition
rows: `WAITING → TODO` when `result.json` lands, `TODO → DONE` after
analysis. One row analyzed per iteration. If no row is ready, exit
quietly (loop will re-fire and check again).

**Phase 3 (final iteration)** — when every row is `DONE` or `FAILED`,
write the summary section (best-by-loss, best-by-walltime, divergers,
stage-2 recommendations) and emit the completion promise.

# Phase 2 — one row per iteration

1. Run `bash scripts/fetch_sweep_results.sh` (rsyncs `~/projects/deqn-jax/runs/sweep_so/` from DGX → local).
2. For every row currently `WAITING`: check whether
   `runs/sweep_so/<run_id>/result.json` exists locally now. If yes,
   bump status to `TODO` (still no analysis written; just transitioned
   from "no data yet" to "ready to analyze").
3. Find the first row with `[STATUS: TODO]`.
   - If none, and `WAITING` rows remain: print `"waiting on N runs"`
     and exit — the loop will re-fire later.
   - If none, and zero `WAITING` rows remain: jump to **Phase 3**.
4. Read `runs/sweep_so/<run_id>/result.json`. Extract:
   - `status` (ok / error)
   - `final_loss`, `final_grad_norm`
   - `wall_seconds`
   - `final_residuals` (per-equation residual mean dict)
   - For `error` runs: `error_type`, `error_msg`
5. Optionally peek at TB events for the loss curve shape
   (`runs/sweep_so/<run_id>/tb/`):
   - Use `python -c "from tensorboard.backend.event_processing import event_accumulator; ea = event_accumulator.EventAccumulator('runs/sweep_so/<run_id>/tb'); ea.Reload(); ..."`
     to extract `train/loss` series.
   - Characterize: monotone decrease? plateau early? blowup? (1-line note)
6. Update the row in the results table:
   - `status` → `DONE` (or `FAILED` if status was `error`)
   - `final_loss` → `<value>` (or `—` for failures)
   - `wall (s)` → `<int>`
   - `notes` → 1-sentence qualitative summary (what the loss curve did
     OR the error class for failures, e.g.
     `"NaN at ep ~600 — damping too low?"`,
     `"smooth log-linear decline"`,
     `"plateau at 1e-3 by 1k eps"`)
7. Update the `## Status` counts at the top.
8. `git add docs/sweeps/disaster_second_order/results.md` (NOT `git add -A` — sibling sessions
   have contaminated commits before) `&& git commit -m "sweep: analyze <run_id>"` and exit.

# Phase 3 — summary + promise

1. When the status counts show `WAITING=0` and only `DONE`/`FAILED`
   remain, append a `## Summary` section:

   - **Best by final_loss** (top 3) — optimizer + lr + value.
   - **Best by wall-clock** to reach loss ≤ X (X = top quartile median).
   - **Divergers** — anything in FAILED, plus runs that finished but
     with `final_loss > 10x` of the best.
   - **Patterns** — e.g. "all curvature methods (gn/ign/lm) need
     damping above 1e-4", or "muon+1e-3 looks competitive but
     plateaus", or "ngd was only stable at lr=1e-4".
   - **Stage-2 recommendations** — which 2-3 cells deserve seed
     replicates (3 seeds), and whether to vary damping / hidden_size.

2. Update `## Status` counts.
3. `git add docs/sweeps/disaster_second_order/results.md && git commit -m "sweep: finalize summary + stage-2 recs"`.
4. Emit the completion promise:

   <promise>DISASTER SO SWEEP ANALYZED</promise>

# Hooks

  or shell scripts.
- The DGX-side launcher is decoupled from this loop. If it stalls or
  fails wholesale, individual rows will sit in `WAITING` forever; in
  that case manually mark them `BLOCKED` with a reason (and exit).

# Out of scope

- Re-running training (the launcher handles that; this loop only
  reads results).
- Modifying the sweep grid (already locked in
  `scripts/sweep_disaster_second_order.py`).
- Changing the base config.
- Stage 2 (seed replicates / ResMLP) — recommendations only, no
  execution.
